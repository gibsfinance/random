## Circuit 1 — `wordle_clue.circom`

Prover = house. Public: `commit`, `guess[5]`, `clue[5]`. Private: `word[5]`, `salt`.

Constraints:
1. `Poseidon(word[0..4], salt) == commit` — single `Poseidon(6)` call.
2. Range checks: `word[i]`, `guess[i]` ∈ [0,25] and `clue[i]` ∈ [0,2], each via a single
   `LessThan` on `(value, bound)` — since `LessThan` decomposes into bits via `Num2Bits`, an
   out-of-range field element simply can't produce a valid witness (no separate "is it even a
   valid field element in range" check needed).
3. Duplicate-letter-correct scoring, exactly per the task spec's formulation:
   `green[i] = IsEqual(guess[i], word[i])`; `avail_i` = count of `guess[i]` among non-green word
   positions; `usedBefore_i` = count of earlier non-green guess positions with the same letter;
   `yellow_i = (1-green[i]) * LessThan(usedBefore_i, avail_i)`; asserted via
   `clue[i] === 2*green[i] + yellow_i`.

**Constraint count:** 1342 total (577 non-linear + 765 linear, per `circom`'s own report;
`snarkjs r1cs info` reports the final compiled 1342). 11 public inputs, 6 private inputs.
Dev ptau power used: **12** (2^12 = 4096 ≥ 1342).

## Circuit 2 — `sudoku_solve.circom`

Prover = player. Public: `puzzle[81]` (0=blank else 1..9), `commit`. Private: `solution[81]`,
`salt`.

Constraints:
1. Each `solution[i] ∈ [1,9]`: a single `LessThan(4)` on `(solution[i]-1, 9)` — if
   `solution[i]` were 0, `solution[i]-1` becomes `p-1` (a ~254-bit field element) which cannot
   decompose into 4 bits, so witness generation fails; this single check enforces both bounds.
2. `puzzle[i] != 0 ⇒ solution[i] == puzzle[i]`, encoded as
   `IsZero(puzzle[i]) OR IsEqual(solution[i], puzzle[i])` (inclusive-OR via `a+b-a*b === 1`).
3. All 9 rows + 9 columns + 9 boxes (27 groups) are each a permutation of 1..9: for every group
   and every `v ∈ 1..9`, exactly one cell in the group equals `v` (`Σ IsEqual(cell,v) === 1`).
4. Commitment (see scheme below): `Poseidon(rowDigest[0..8], salt) == commit`.

**Constraint count:** 22,462 total (9426 non-linear + 13036 linear per circom; `snarkjs r1cs
info` reports 22462 final). 82 public inputs (81 puzzle cells + commit), 82 private inputs (81
solution cells + salt). Dev ptau power used: **15** (2^15 = 32768 ≥ 22462).

### Commitment scheme (Sudoku) — reproducible spec

81 cells + 1 salt = 82 values exceeds Poseidon's ≤16-input limit, so a two-level sponge is used,
grouped by **row** (cell index = `r*9 + c`):

```
rowDigest[r] = Poseidon(solution[r*9], solution[r*9+1], ..., solution[r*9+8])   for r = 0..8
                                                                    (one Poseidon(9) call each)
commit       = Poseidon(rowDigest[0], ..., rowDigest[8], salt)      (one Poseidon(10) call)
```

This is implemented identically on both sides:
- circuit: `circuits/sudoku_solve.circom`, section "4. commitment" (two Poseidon template
  instantiations, `Poseidon(9)` ×9 then `Poseidon(10)` ×1).
- JS: `src/sudoku.ts` → `sudokuCommit(solution, salt)`, using `circomlibjs`'s `buildPoseidon()`
  with the exact same row-grouping and call order. Test `commit is stable for the same
  solution+salt` in `test/sudoku_solve.test.ts` exercises this function directly; the
  proof-based tests exercise the circuit side and cross-check equality (`Poseidon(...) ==
  commit` is a circuit constraint, so any mismatch fails witness generation, which the "wrong
  commit" negative test confirms).

This grouping was chosen (over e.g. flat 16-cell chunks) because it's self-documenting (one
hash per Sudoku row) and keeps both the circuit and the JS mirror trivial to eyeball for
correctness.

## Off-chain proving harness (`src/harness.ts`)

Reusable across both circuits:
1. `compileCircuit(name)` — shells out to the `circom` binary (`-l node_modules` so
   `include "circomlib/circuits/..."` resolves), caching `build/<name>/{name.r1cs, name_js/name.wasm}`.
2. `ensurePtau(power)` — dev powers-of-tau via the `snarkjs` CLI: `powersoftau new bn128
   <power>` → one `contribute` (fixed dev entropy string) → `prepare phase2`. Cached per power
   under `build/pot<power>_final.ptau` and reused across both circuits (both fit under a single
   ptau sized for the larger — sudoku — circuit if you want to share one file; the tests use
   power 12 for wordle and power 15 for sudoku, each cached independently).
3. `setupCircuit(name, ptauPower)` — `snarkjs groth16 setup` → `<name>_final.zkey`, then
   `zkey export verificationkey` → `<name>_vkey.json`. **This zkey is a single, deterministic,
   dev-only contribution — it is explicitly NOT a production trusted setup** (no
   randomness-destroying final contribution/beacon). Fine for proving the circuit logic
   off-chain, which is this milestone's whole point; a real ceremony is an M1 concern.
4. `prove(setup, input)` / `verify(setup, publicSignals, proof)` — thin wrappers over
   `snarkjs.groth16.fullProve` / `.verify`, run in-process (no CLI shelling for the hot path).

All generated artifacts live under `examples/games/zk-skill/build/` (gitignored, not committed).

## Test results

`npm test` (`vitest run`) in the package: **12 passed, 0 failed** (2 test files).

- `test/wordle_clue.test.ts` (6 tests):
  - fixed vector `word="crane"` → `[2,17,0,13,4]`, `guess="eerie"` → `[4,4,17,8,4]`
  - JS reference scorer matches the hand-verified expected clue `[grey,grey,yellow,grey,green]`
    = `[0,0,1,0,2]` (only one 'e' in "crane", already claimed by the green at position 4, so
    both extra 'e's in "eerie" are grey; 'r' present but misplaced → yellow)
  - prove+verify succeeds for the honest clue
  - **negative:** tampered clue (one trit flipped) → `prove()` rejects (witness generation
    throws, `ERROR: 4 Error in template WordleClue... `)
  - **negative:** wrong `commit` (+1) → `prove()` rejects
  - commitment determinism check
- `test/sudoku_solve.test.ts` (6 tests):
  - fixed vector: a known-valid, fully solved 9×9 grid (band-rotation construction) with a
    puzzle that reveals only row 0 + column 0 as given clues
  - sanity check that the fixed vector is genuinely valid (JS-side `isValidSolution`)
  - prove+verify succeeds for the valid solution
  - **negative:** a solution edit at a *blank* puzzle cell that duplicates a value in its
    row/box/column → breaks the permutation constraint → `prove()` rejects
  - **negative:** a solution edit at a *given-clue* puzzle cell → breaks the
    agreement-with-clue constraint → `prove()` rejects
  - **negative:** wrong `commit` (+1) → `prove()` rejects
  - commitment determinism check

`npx tsc --noEmit` also passes clean (ambient `.d.ts` added for `snarkjs`/`circomlibjs`, which
ship no type declarations).

Timing: cold run (circuit compile + fresh dev ptau + groth16 setup for both circuits) is ~8
minutes total, dominated by the sudoku ptau/zkey generation (power-15, ~22.5k constraints,
~7 min). Once `build/` is populated, the full suite reruns in ~3 seconds (setup is
cached/skipped when the r1cs/wasm/zkey/vkey files already exist).

  untracked files; `build/` is gitignored).

## Concerns / risk notes

- The `-l node_modules` pnpm resolution for `circomlib` include paths is a bit implicit (relies
  on pnpm hoisting `circomlib` into the package's local `node_modules`, which pnpm workspaces do
  for direct deps) — if the workspace's pnpm hoisting policy changes this could break; worth
  double-checking after any future `pnpm-workspace.yaml` edits.
- circom itself had to be built from source via cargo (no arm64 macOS release binary exists
  upstream as of `v2.2.3`) — CI or other contributors on Apple Silicon will hit the same gap and
  need the same `cargo install` step (~1-2 min build) unless a prebuilt binary is vendored or CI
  runs on x86/linux.
